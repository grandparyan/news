什麼是常見編碼？一篇搞懂明文、Base64 與雜湊

發布日期：2025-11-19 | 分類：網路基礎

在網路世界中，資料很少以其最原始的形態傳輸。為了效率、相容性或安全性，資料會被「編碼」(Encoding) 或「雜湊」(Hashing)。

然而，許多初學者會將「編碼」與「加密」(Encryption) 混淆。這篇文章將介紹幾種最常見的資料表示法：明文、Base64 編碼，以及常用於「Auth」(驗證) 的雜湊。

明文 (Plain Text)

「明文」是最簡單的資料形式，它**沒有經過任何編碼或加密**，是人類可以直接閱讀的文字。

  • 範例： hello、我的密碼是123456。
  • 優點： 易於閱讀和除錯。
  • 缺點： **極度不安全**。如果密碼、信用卡號等敏感資料以明文形式儲存或傳輸（例如在 Telnet 或 FTP 中），任何竊聽者都能輕易讀取。

Base64 編碼

Base64 是一種「編碼」，**它不是加密**。編碼的目的是為了「相容性」，而不是「安全性」。

有些系統（例如 Email）最初只設計來傳輸 ASCII 文字。如果您想傳輸圖片、聲音或任何二進位 (binary) 檔案，就會遇到問題。Base64 的作用，就是將這些二進位資料轉換成 64 個「安全」的 ASCII 字元（A-Z, a-z, 0-9, +, /）組合，使其可以在任何系統上傳輸。

  • 範例： 將 Hello 進行 Base64 編碼，會得到 SGVsbG8=。
  • 重點： 任何人都可以將 SGVsbG8= **反向解碼**回 Hello。
  • 切勿用 Base64 "加密" 密碼！ 這就像用透明膠帶把鑰匙貼在門上，毫無安全性可言。
# 編碼
"Hello"  -- (Base64 Encode) --> "SGVsbG8="

# 解碼
"SGVsbG8=" -- (Base64 Decode) --> "Hello"

釐清「Auth」：雜湊 (Hashing) 與驗證

您提到的 `Auth` (Authentication，身分驗證) 並非一種編碼。但在驗證過程中，我們會使用「雜湊」來保護資料，有時也會使用 Base64 來傳輸資料。

1. 雜湊 (Hashing)：用於儲存密碼 (Auth)

雜湊 (Hash) 是一種**單向**的數學運算。它會將任意長度的輸入（例如您的密碼）轉換成一個固定長度的「雜湊值」（或稱「指紋」）。

  • 範例 (MD5 雜湊)：
    password123 -- (MD5) --> e10adc3949ba59abbe56e057f20f883e
  • 關鍵特性：
    1. **單向性：** 您**無法**從雜湊值 e10... 逆向推算出原始密碼 password123。
    2. **唯一性：** 只要輸入稍微不同（例如 Password123），產生的雜湊值就會完全不同。
  • 用途： 當您註冊網站時，系統**不應該**儲存您的「明文密碼」，而應該儲存密碼的「雜湊值」。當您登入時，系統會將您輸入的密碼再次雜湊，並比對雜湊值是否相符。這樣一來，即使資料庫被盜，駭客也只能拿到一堆無法逆向的雜湊值，而不是您的明文密碼。

2. HTTP Basic Auth：使用 Base64 的 Auth

這可能是 `Auth` 和 `Base64` 經常被搞混的原因。HTTP 基本驗證 (Basic Auth) 是一種簡單的網頁驗證機制。

它會將您的「使用者名稱」和「密碼」用冒號 : 連接起來（例如 user:password123），然後對這個字串進行 **Base64 編碼**，並將其放在 HTTP 標頭 (Header) 中傳送。

  • 範例：
    user:password123 -- (Base64 Encode) --> dXNlcjpwYXNzd29yZDEyMw==
  • 安全警告： 如前所述，Base64 是可逆的。駭客只要竊聽到這段 Base64 字串，就能馬上解碼回明文的帳號密碼。因此 Basic Auth 除非搭配 HTTPS，否則也極不安全。
